## 非阻塞建议

- 建议在实现阶段把“宿主 schema 已预先声明显式写回流程”再落成一句更强的操作语：**未命中显式流程时，一律不得因 query 自行写回**，减少执行时自由解释空间。
- 建议在 `references/workflows.md` 中把 query 写回补成一个简短判定清单（是否用户明确要求 / 是否命中宿主显式流程 / 是否需要同步记账），便于后续自检。
- 建议把 `log.md` 的 query 记录条件再写得更紧一点，例如仅记录“发生了授权写回的 query”，避免把普通只读问答也写进日志导致膨胀。

## 修复指导

无需再改 RFC 才能进入实现；实现时只需保持以下最小约束不漂移：

1. `query` 默认严格只读。
2. 非用户明确要求时，只有命中宿主 schema 的**显式写回流程**才可写回。
3. 一旦写回，必须同步处理目标页面、`index.md`（若新增页）与 `log.md`。
4. 不得把“结构化产物”“高价值内容”等主观判断当作授权来源。

- **可实现**：**满足**。执行者已能据此区分 query 的只读路径与授权写回路径。
- **可验证**：**满足**。可用样例验证“默认不写回”和“命中显式授权时才写回”两种行为。
