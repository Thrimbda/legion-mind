# RFC 审查报告

## 结论
PASS

本次更新已经补上此前两个阻塞点：

- 命令兼容矩阵已把保留命令、删除命令、最小返回字段和统一错误语义定死，设计现在**可实现、可验证**。
- 回滚章节已覆盖“旧文件仍存在”和“被新 CLI init 后再回退”两类场景，并给出最小 stub 恢复步骤，设计现在**可回滚**。

整体上，RFC 已经收敛到足够小且边界明确的方案，可以进入实现。

- [ ] 无

## 非阻塞建议

- **`status` 返回字段里的 `currentTask` 建议删掉或改名。**
  - 现在兼容矩阵写 `status` 成功返回 `taskId`, `name`, `currentTask`, `progress.*`，但同一 RFC 又明确“不再隐式读取 currentTask”。
  - 这会让 reviewer 怀疑你是不是又保留了一个“当前任务”概念。
  - 最小化复杂度建议：直接删掉 `currentTask` 字段；若只是想表达“当前查询的任务”，`taskId` 已经够了。

- **单任务命令的“按需文件校验”可以再少一点歧义。**
  - 例如 `review list/respond` 现在写“需要对应 raw docs”，但没有像 `plan/log/tasks` 那样精确定义最小文件边界。
  - 最小化复杂度建议：实现时只校验命令实际触达的目标文件，不额外发明统一 review 目录规则。

- **未决问题建议在实现前就收口。**
  - `task list` 是否保留派生 `status` 字段，目前 RFC 倾向“不保留”，这与最小方案一致。
  - 最小化复杂度建议：直接在 RFC 中定为“只返回 progress，不返回 status”，避免实现时再做兼容性妥协。

## 修复指导

无需重写 RFC；按最小改动补齐即可：

1. 把 `status` 返回中的 `currentTask` 删掉或改成无歧义字段。
2. 若要继续保守收敛，可顺手把 `task list` 的未决 `status` 字段直接定为不保留。
3. `review` 相关文件校验保持最小按需，不要再扩出新的隐藏 schema。

该 RFC 已满足进入实现的最低条件：
