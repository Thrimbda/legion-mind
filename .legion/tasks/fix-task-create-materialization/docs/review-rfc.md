# RFC 审查报告

## 结论
PASS

该 RFC 总体收敛，方案范围小、目标明确，且用 staging 目录后整体 rename 的办法，确实比“直接写最终目录 + 失败回滚”更简单也更可靠，符合这次任务的最小修复原则。

同时，本设计满足：
- **可实现**：改动集中在 `writeTaskDraft()` 物化路径，外部 CLI 接口不变。
- **可验证**：已覆盖成功路径、写入中途失败、最终落位失败三类关键场景。
- **可回滚**：可回退到原直接写入策略，兼容性面较小；遗留 `.tmp-*` 可手工清理。

## 阻塞问题
- [ ] 无

## 非阻塞建议
- 明确 **rename 竞争失败** 的错误语义：若在预检查后最终目录被其他进程抢先创建，建议尽量收敛为稳定、可理解的冲突错误（例如继续落到“已存在/冲突”语义），避免出现过于底层的系统错误直接冒泡。
- 把 **故障注入方式** 再写具体一层：建议在 RFC 中点明最小验证手段（例如对内部 helper 做 stub/mock，或对 `writeTaskDraft()` 的文件写入/rename 边界做单测注入），避免实现阶段为了测试而临时扩散接口。
- 在回滚小节补一句 **回滚后可接受残留**：即使宿主级中断留下 `.tmp-*`，它们不会被 `listTasks()` 当成合法任务；这样能更明确说明为什么该残留是可容忍的，而不是新的数据损坏。

## 修复指导
1. 保持当前推荐路线，不要扩展到 schema、status、task list 或额外维护命令。
2. 实现时把复杂度锁死在 `writeTaskDraft()`：
   - 创建 staging root
   - 在 staging 内写完 `docs/`、`plan.md`、`log.md`、`tasks.md`
   - 最后一次性 rename 到最终目录
3. 出错时仅做 best-effort 清理 staging；不要为了“绝不残留 tmp”引入后台清扫、重试循环或额外状态机。
4. 验证时最少覆盖：
   - 成功创建后立即 `status` / `task list` 可读
   - staging 写入失败时最终目录不存在
   - 最终 rename 失败时最终目录不存在或不可见为半成品

## 审查说明
- 该设计的核心优点是：把“最终目录过早可见”这个结构性问题，收敛为“只有 staging 目录可见，最终目录只在完整时出现”。
- 对这类 one-off 且未稳定复现的问题，先做最小防御性修复是合理的；继续追唯一根因反而更可能扩大范围。
- 当前剩余 caveat 是：**宿主级中断后可能遗留 `.tmp-*` 目录**。但这些目录不进入合法 taskId 集，也不影响 `status` / `task list` 的正确性，因此目前可接受。
