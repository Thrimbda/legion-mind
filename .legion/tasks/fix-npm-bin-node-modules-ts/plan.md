# Fix npm bin TypeScript runtime failure

## Task contract

- **Name:** Fix npm bin TypeScript runtime failure
- **Task ID:** `fix-npm-bin-node-modules-ts`
- **Goal:** publish a patch release that makes `npx lgmind ...` work from an installed npm package.
- **Problem:** `lgmind@0.2.0` package bins execute TypeScript files from inside `node_modules`. Node's type stripping rejects that with `ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`, so `npx lgmind install` fails before the CLI can run.

## Acceptance

- npm package bin entrypoints no longer import or execute `.ts` files from `node_modules`.
- Packed/installed package execution is covered by regression tests, not only repo-local source execution.
- `npx`-equivalent command execution works for `lgmind --version`, `lgmind setup --agent opencode --help` or an equivalent non-destructive command, and `setup-opencode --help` from an installed package layout.
- Package version is bumped to `0.2.1`.
- Release PR is merged, GitHub Actions trusted-publishing workflow publishes `lgmind@0.2.1`, and npm `latest` resolves to `0.2.1`.
- Task evidence, walkthrough, and wiki writeback are recorded.

## Scope

- Fix runtime packaging/bins for npm execution.
- Add generated or maintained JavaScript runtime files needed by the npm package.
- Update package allowlist to publish runnable JavaScript runtime files.
- Add regression coverage that installs or extracts a packed package into a `node_modules`-like location and executes package bins.
- Bump patch version and publish after PR merge.

## Non-goals

- No new user-facing CLI behavior beyond making the published behavior runnable.
- No rewrite of setup lifecycle semantics.
- No replacement of the current TypeScript source/test development flow unless required for runtime packaging.
- No local npm token or OTP workaround.

## Assumptions

- Patch version `0.2.1` is appropriate because `0.2.0` introduced a runtime packaging bug.
- Source `.ts` files can remain for development/tests, but published bin wrappers must use runnable `.js` files.
- The existing GitHub Actions trusted-publishing workflow can publish the patch after the package version is bumped and merged.

## Constraints

- Work happens in `.worktrees/fix-npm-bin-node-modules-ts/` on branch `legion/fix-npm-bin-node-modules-ts`.
- Base ref is `origin/master`.
- Use repo-local npm cache: `npm_config_cache=.cache/npm`.
- Before pushing, run `git fetch origin && git rebase origin/master` in the worktree.
- Publish only from the merged `master` baseline through the trusted-publishing workflow.

## Risks

- Runtime JS generated from TS may drift if not covered by tests.
- Pack tests that only inspect file lists will miss this bug; regression must execute from a package-like installed location.
- Publishing before verifying installed-package execution would repeat the `0.2.0` failure mode.

## Recommended path

- Keep TypeScript source for development.
- Add committed JavaScript runtime files for published package bins, generated by stripping TypeScript syntax from the existing runtime source.
- Point bin wrappers at `.js` runtime files, include those files in `package.json#files`, and keep `.ts` source in the package only if useful for source visibility.
- Add a regression that packs the package, installs/extracts it under a temporary `node_modules/lgmind`, and executes the installed bin wrappers with plain Node.

## Phases

1. Open isolated hotfix worktree and materialize contract.
2. Inspect current bin/runtime packaging path and implement JS runtime fix.
3. Add installed-package regression and bump `package.json` to `0.2.1`.
4. Run regression suite and package dry-run.
5. Review, walkthrough, wiki writeback.
6. Commit, rebase, push PR, merge, cleanup, refresh main.
7. Dispatch trusted-publishing workflow and verify npm `latest = 0.2.1`.
