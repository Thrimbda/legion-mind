# 工作流

## ingest

### 触发

- 用户提供新 source 并要求处理；或宿主流程要求定期摄入 raw sources。

### 标准步骤

1. 读取 source，提取关键事实、主张、时间、角色、数据与不确定点。
2. 判断应更新哪些 wiki 页面：
   - 来源摘要页；
   - 已有主题/实体/项目页；
   - 必要时新增综合页或待验证页。
3. 将新信息合并到相关页面，保留证据指向；新旧信息冲突时显式标记争议或 superseded，而不是静默覆盖。
4. 更新 `index.md`：为新增页面补入口；为重要页面维护一句话说明。
5. 向 `log.md` 追加一次 ingest 记录，只写 source/page ID、动作摘要与待验证项，不默认复制敏感标题或摘录。

### 最小产物

- 至少 1 个来源摘要或来源页。
- 至少 1 处对既有 wiki 页面更新或新增页面。
- `index.md` 已同步。
- `log.md` 已追加，标题建议：`## [YYYY-MM-DD] ingest | <source-id-or-safe-title>`。

## query

### 默认模式

- 严格只读。
- 先读 `index.md`，再进入相关页面综合答案。
- 优先复用已有综合结果；缺口部分再读相关页或必要 source。
- 除非为了补证据，不要退化为全量 source 扫描。

### 输出要求

- 基于 wiki 回答，并指出关键依据页面。
- 当页面间存在冲突时，说明冲突与当前证据状态。
- 若 wiki 无法支持结论，明确说明缺失，而不是编造补全。

### 允许写回的唯一条件

仅当同时满足以下条件时，才允许把 query 结果写回 wiki：

1. 用户明确要求把结果沉淀为页面、卡片或维护项；
2. 宿主 schema 已显式声明该类 query 的写回流程，且同时规定：
   - 哪类操作者或维护者允许执行写回；
   - 触发条件；
   - 目标页面类型或落点；
   - 允许写入的字段或变更范围；
   - `index.md` / `log.md` 如何同步。

若任一条件缺失，query 仍停留在只读路径；最多输出建议的沉淀方案或把工作升级为后续 ingest / maintenance 任务，不直接写回。

### 写回后的必做动作

- 新增或更新目标页面。
- 仅在新增、重命名、合并、删除、分类变化，或目录描述已不准确时更新 `index.md`。
- 向 `log.md` 追加一条 query 或 analysis 记录。
- 在日志中记录授权依据、目标页面 ID、涉及的 source/page ID 与安全摘要，不默认写入敏感标题、原文摘录或访问路径。

## lint

### 目标

定期健康检查 wiki，而不是直接重写一切。

### 检查清单

- 页面之间是否存在冲突结论；
- 是否有结论已被较新 source 推翻或削弱；
- 是否存在孤儿页；
- 是否缺失应有互链；
- 是否有重要概念被多处提及却没有独立页面；
- 是否存在证据缺口或待补 source。

### 执行原则

- 证据不足时，不直接改写结论；先标记问题与建议。
- 优先产出可执行修复建议：补链接、补页、补来源、降级断言强度、增加“待验证”。
- 当宿主 schema 有维护页或 backlog 页约定时，按宿主约定落地。
- 向 `log.md` 追加一条 lint 记录，只记录检查范围、结论摘要与后续动作，不把全部问题明细直接倒入日志。

### 最小输出

- 问题清单。
- 每个问题的影响范围与建议动作。
- 已追加的 lint 日志，标题建议：`## [YYYY-MM-DD] lint | <scope-or-safe-id>`。
